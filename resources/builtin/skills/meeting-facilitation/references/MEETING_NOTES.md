# 会议记录最佳实践

本文档说明如何高效记录会议内容，提取关键信息，转化为可执行的行动。

## 会议记录的目的

**不是逐字记录，而是：**
1. 记录决策及其理由
2. 提取行动项和责任人
3. 保留关键讨论点
4. 便于未参会者了解会议结果

## 实时记录技巧

### 速记符号

**常用符号：**
```
✅ 决策/确定
❓ 待确认/存疑
⚠️ 风险/问题
💡 想法/洞察
👤 人名/责任人
📅 日期/截止时间
🔗 链接/参考资料
→ 导致/因此
∴ 所以/结论
vs 对比
```

**示例：**
```
[10:15] 讨论新功能上线时间
- 张三: 建议6.15 ✅
- 李四: 担心时间太紧 ⚠️
- 决策: 6.20上线 ✅ (留5天缓冲)
- 行动: 李四 更新项目计划 📅6.12前
```

### 关键词捕捉

**触发记录的关键词：**
```
决策类：
- "我们决定..."
- "最终方案是..."
- "确定采用..."

行动类：
- "XX负责..."
- "我会在XX前完成..."
- "需要XX提供..."

问题类：
- "还需要确认..."
- "存在风险..."
- "这个问题待解决..."
```

### 分角色记录

**记录不同人的观点：**
```
议题：选择技术方案

[张三/技术负责人]
- 推荐方案A
- 理由：成熟稳定，社区活跃
- 担心：学习成本高

[李四/产品经理]
- 倾向方案B
- 理由：开发效率高，快速上线
- 担心：长期维护成本

[决策]
- 采用方案A
- 原因：长期来看更稳定
- 妥协：安排内部培训降低学习成本
```

## 行动项提取

### 行动项的5个要素

**完整的行动项必须包含：**
```
1. What（做什么）：具体任务
2. Who（谁做）：责任人（唯一）
3. When（什么时候）：截止时间
4. Why（为什么）：任务目的（可选）
5. How（怎么验证）：完成标准

示例：
❌ "优化性能"（太宽泛）
✅ "将首页加载时间从3秒优化到1.5秒，由张三负责，6月15日前完成，验收标准：Lighthouse评分>90"
```

### 自动识别行动项

**触发词识别：**
```
明确承诺：
- "我来负责XX"
- "我会在XX前完成XX"
- "这个由我跟进"

需求表达：
- "需要XX部门提供XX"
- "希望XX能协助XX"
- "请XX确认XX"

待办事项：
- "这个需要确认"
- "我们回去核实"
- "下周一提交"
```

### 行动项状态管理

**状态定义：**
```
📝 待开始（To Do）
  - 已分配但未开始
  - 可能有依赖未完成

⏳ 进行中（In Progress）
  - 正在执行
  - 进度可追踪

⚠️ 有风险（At Risk）
  - 可能延期
  - 遇到阻塞

✅ 已完成（Done）
  - 已交付
  - 已验收

🚫 已取消（Cancelled）
  - 不再需要
  - 已变更需求
```

## 不同类型会议的记录要点

### 1. 日常站会（Daily Standup）

**时长：** 5-15分钟
**参与：** 项目团队

**记录重点：**
```markdown
## [日期] 站会记录

### 张三
- ✅ 昨天：完成了用户登录功能
- 📝 今天：开发注册功能
- ⚠️ 阻塞：等待设计稿确认

### 李四
- ✅ 昨天：修复了3个Bug
- 📝 今天：继续测试支付模块
- ⚠️ 阻塞：无

### 待跟进
- [ ] 设计稿确认 - @王五设计师
```

**特点：**
- 简洁（每人1-2分钟）
- 聚焦今天要做的事
- 识别阻塞

### 2. 项目评审会

**时长：** 1-2小时
**参与：** 跨部门

**记录重点：**
```markdown
## 项目评审会纪要

### 评审内容
- 项目名称：XX系统改造
- 评审版本：V1.0
- 评审人员：产品/技术/测试/运营

### 评审意见

#### 功能完整性 ✅
- 核心功能已实现
- 符合需求文档

#### 技术方案 ⚠️
- 建议：增加缓存层（李四）
- 建议：优化数据库查询（张三）

#### 测试覆盖 ❌
- 问题：单元测试覆盖率仅60%
- 要求：提升到80%以上
- 负责：张三，6.20前

### 评审结论
❌ 不通过，需修改后重新评审
✅ 通过，可以上线
⚠️ 有条件通过，解决XX问题后上线

### 后续行动
[行动清单]
```

### 3. 头脑风暴会

**时长：** 30-60分钟
**参与：** 创意团队

**记录重点：**
```markdown
## 头脑风暴：XX产品定位

### 规则
- 不批判，只记录
- 鼓励大胆想法
- 在别人想法基础上发散

### 产生的想法

#### 类别1：目标用户
1. 想法：专注大学生群体
   - 提出人：张三
   - 支持：⭐⭐⭐⭐（4票）
   - 理由：市场空白，需求明确

2. 想法：面向职场新人
   - 提出人：李四
   - 支持：⭐⭐（2票）

#### 类别2：核心功能
[同上]

### 筛选结果
- ✅ 立即执行：想法1、想法5
- 📋 列入储备：想法2、想法6
- ❌ 暂不考虑：想法3（理由：资源限制）

### 下一步
[将入选想法转化为具体方案]
```

**特点：**
- 不过滤想法（记录阶段）
- 分类整理
- 投票筛选

## 会议记录的常见错误

### ❌ 错误1：流水账

**错误示例：**
```
张三说了XXX
李四说了XXX
王五说了XXX
```

**为什么错：**
- 信息量大，没有重点
- 未提炼关键决策
- 不利于后续查阅

**正确做法：**
```
讨论议题：XX
核心观点：
- 正方（张三、李四）：观点A，理由X
- 反方（王五）：观点B，理由Y
最终决策：采用观点A，原因：Z
```

---

### ❌ 错误2：无行动项

**错误示例：**
```
会议讨论了XX问题，大家一致认为需要改进。
```

**为什么错：**
- 没有具体行动
- 没有责任人
- 没有截止时间
- 会议=白开

**正确做法：**
```
决策：改进XX
行动项：
- [ ] 张三负责调研方案（6.15前）
- [ ] 李四负责技术评估（6.18前）
- [ ] 王五负责成本核算（6.18前）
```

---

### ❌ 错误3：记录不及时

**错误示例：**
会议结束3天后才发纪要

**为什么错：**
- 记忆模糊，遗漏重要信息
- 行动项延误
- 参会者无法及时跟进

**正确做法：**
- 会议当天（最晚24小时内）发纪要
- 实时记录，会后整理
- 关键决策当场确认

## 总结

高效会议记录的要点：
1. **实时记录**：会中记录，会后整理
2. **提取关键**：决策、行动、问题
3. **明确责任**：每个行动项有唯一负责人
4. **及时发布**：24小时内发送纪要
5. **跟进落实**：定期检查行动项完成情况
