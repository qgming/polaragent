# 任务拆解方法论

本文档详细说明如何科学地拆解复杂任务，从"不知从何下手"到"清晰可执行"。

## WBS 工作分解结构

### 什么是 WBS

**定义：** Work Breakdown Structure，将项目目标逐层分解为可管理的工作单元。

**核心思想：**
```
大问题 → 拆解为 → 小问题 → 逐个解决
```

### 分解维度

**维度1：按阶段分解**
```
产品开发项目
├─ 需求阶段
├─ 设计阶段
├─ 开发阶段
├─ 测试阶段
└─ 上线阶段
```

**维度2：按交付物分解**
```
产品开发项目
├─ 需求文档
├─ 设计稿
├─ 前端代码
├─ 后端代码
└─ 测试报告
```

**维度3：按功能模块分解**
```
电商系统
├─ 用户模块（注册/登录/个人中心）
├─ 商品模块（浏览/搜索/详情）
├─ 订单模块（下单/支付/物流）
└─ 后台模块（商品管理/订单管理）
```

### 分解原则：MECE

**MECE = Mutually Exclusive, Collectively Exhaustive**
- Mutually Exclusive（相互独立）：任务之间不重叠
- Collectively Exhaustive（完全穷尽）：所有任务加起来=完整项目

**示例：**
```
✅ 符合 MECE：
网站开发
├─ 前端开发
├─ 后端开发
└─ 测试

❌ 不符合 MECE：
网站开发
├─ 前端开发
├─ JavaScript 开发（与前端重叠）
└─ 测试
```

### 分解粒度

**粒度原则：**
- 最底层任务：1-5 天可完成
- 能明确责任人
- 有清晰的交付物
- 能独立验收

**示例：**
```
❌ 粒度过大（难以执行）：
- 完成产品开发（数月工作，无法把控）

❌ 粒度过小（管理成本高）：
- 写一个函数（数小时工作，管理混乱）

✅ 粒度适中（可管理）：
- 完成用户注册模块（2-3天，交付物明确）
```

```
用户认证系统
├─ 注册功能
├─ 登录功能
├─ 密码找回
└─ 个人信息管理
```

**第2层：细分任务**
```
注册功能
├─ 前端页面
│  ├─ 注册表单（邮箱/密码/验证码）
│  ├─ 表单验证（前端校验）
│  └─ 错误提示（提示信息设计）
├─ 后端接口
│  ├─ 注册 API（POST /api/register）
│  ├─ 邮箱验证码发送
│  ├─ 用户数据存储（数据库设计）
│  └─ 密码加密（bcrypt）
└─ 测试
   ├─ 单元测试（接口测试）
   ├─ 集成测试（前后端联调）
   └─ 用户测试（真实场景验证）
```

**第3层：开发任务**
```

├─ 任务1：设计表单 UI（1天）
│  └─ 交付物：Figma 设计稿
├─ 任务2：实现 HTML 结构（0.5天）
│  └─ 交付物：可运行的表单页面
├─ 任务3：实现表单验证（1天）
│  ├─ 邮箱格式验证
│  ├─ 密码强度验证（8位+大小写+数字）
│  └─ 验证码倒计时
└─ 任务4：接口对接（0.5天）
   └─ 提交表单并处理响应
```

## 估算工作量

### 三点估算法

**公式：**
```
预期时间 = (乐观时间 + 4×最可能时间 + 悲观时间) / 6
```

**示例：**
```
任务：开发用户注册页面

乐观时间：2天（一切顺利）
最可能时间：3天（正常情况）
悲观时间：5天（遇到困难）

预期时间 = (2 + 4×3 + 5) / 6 = 3.2 天 ≈ 4天
```

### 历史数据法

**记录实际耗时：**
```
任务类型：表单页面开发
历史数据：
- 登录页：实际3天
- 注册页：实际4天
- 找回密码页：实际2.5天
平均耗时：3.2天

下次估算：
类似表单页面 = 3.2天 × 1.2（缓冲） = 约4天
```

## 任务依赖关系

### 依赖类型

**1. 前置依赖（FS: Finish-to-Start）**
```
A 完成后，B 才能开始

示例：
设计稿完成 → 前端开发开始
```

**2. 同步启动（SS: Start-to-Start）**
```
A 开始后，B 才能开始

示例：
开发启动 → 测试环境准备启动
```

**3. 同步结束（FF: Finish-to-Finish）**
```
A 完成时，B 也要完成

示例：
开发完成 → 单元测试完成
```

### 识别关键路径

**关键路径 = 最长的依赖链**

**示例：**
```
任务A（3天） → 任务C（5天） → 任务E（2天） = 10天 ← 关键路径
任务B（4天） → 任务D（3天） → 任务E（2天） = 9天

关键路径：A → C → E（10天）
项目最短完成时间：10天
```

**优化策略：**
- 优先保证关键路径任务
- 关键路径任务延误 = 整个项目延误
- 非关键路径任务有缓冲空间

## 总结

任务拆解的要点：
1. **自上而下**：从目标开始，逐层细化
2. **MECE 原则**：相互独立、完全穷尽
3. **适当粒度**：1-5天可完成的工作单元
4. **明确交付**：每个任务都有清晰的产出
5. **识别依赖**：梳理任务间的先后关系
6. **三点估算**：用历史数据和经验估算时间
