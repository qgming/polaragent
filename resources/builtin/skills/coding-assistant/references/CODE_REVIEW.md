# 代码审查清单

本文档提供系统的代码审查检查项，确保代码质量、安全性和可维护性。

## 审查维度

### 1. 功能正确性

**核心问题：代码是否实现了需求？**

```
✓ 检查项：
□ 是否实现了所有需求？
□ 边界条件是否处理？
  - 空值（null/undefined/None）
  - 空数组/空对象
  - 零值
  - 负数
  - 极大值/极小值
□ 错误场景是否处理？
  - 网络错误
  - 权限不足
  - 数据格式错误
□ 逻辑分支是否完整？
  - if-else 是否覆盖所有情况
  - switch-case 是否有 default
```

**示例：**
```javascript
// ❌ 差的实现（未处理边界）
function getFirstItem(array) {
  return array[0];  // 如果 array 为空？为 null？
}

// ✅ 好的实现
function getFirstItem(array) {
  if (!array || array.length === 0) {
    return null;  // 或 undefined，或抛出错误
  }
  return array[0];
}
```

### 2. 代码可读性

**核心问题：其他人能否快速理解？**

```
✓ 变量命名：
□ 是否使用有意义的名称？
  ❌ a, temp, data, result
  ✅ userId, customerList, totalPrice
□ 是否遵循命名规范？
  - camelCase: getUserById
  - PascalCase: UserService
  - UPPER_SNAKE_CASE: MAX_RETRY_COUNT
□ 布尔值是否用 is/has/can 开头？
  ✅ isActive, hasPermission, canEdit

✓ 函数设计：
□ 函数名是否表达意图？
  ❌ process(), handle(), doSomething()
  ✅ calculateTotalPrice(), validateEmail()
□ 函数是否单一职责？
  - 一个函数只做一件事
  - 函数长度 <50 行（建议）
□ 参数数量是否合理？
  - ≤3 个参数（理想）
  - >5 个考虑用对象封装

✓ 注释：
□ 是否解释了「为什么」而非「做什么」？
  ❌ // 循环数组（废话）
  ✅ // 使用逆序遍历，避免删除元素时索引错乱
□ 复杂逻辑是否有注释？
□ 是否删除了无用注释？
```

---

### 3. 代码结构

```
✓ DRY 原则：
□ 是否有重复代码？
  - 相同代码出现 2+ 次 → 提取函数
  - 相似逻辑 → 提取公共部分
□ 是否有魔法数字？
  ❌ if (status === 1)  // 1 是什么意思？
  ✅ const ACTIVE = 1; if (status === ACTIVE)

✓ 单一职责：
□ 每个函数是否只做一件事？
□ 每个类是否只有一个变化的理由？

✓ 依赖注入：
□ 是否硬编码了依赖？
  ❌ function process() { const db = new Database(); }
  ✅ function process(db) { ... }
```

---

### 4. 性能考虑

```
✓ 算法复杂度：
□ 是否有不必要的嵌套循环？
  ❌ O(n²) 可以优化为 O(n)
□ 是否可以用哈希表优化查找？
  ❌ array.find() 每次都遍历
  ✅ 预先建立 Map，O(1) 查找

✓ 数据库查询：
□ 是否有 N+1 查询问题？
□ 是否可以批量查询？
□ 是否添加了必要的索引？

✓ 缓存：
□ 重复计算是否可以缓存？
□ 是否有内存泄漏风险？
```

---

### 5. 安全检查

```
✓ 输入验证：
□ 所有用户输入是否验证？
  - 类型检查
  - 长度限制
  - 格式校验
  - 白名单验证
□ 是否防止 SQL 注入？
  ✅ 使用参数化查询
□ 是否防止 XSS？
  ✅ 转义用户输入
□ 是否防止 CSRF？
  ✅ 使用 CSRF Token

✓ 权限控制：
□ 是否检查用户权限？
□ 是否有横向越权风险？
  - 用户 A 能否访问用户 B 的数据？

✓ 敏感信息：
□ 是否有密码/密钥硬编码？
□ 是否正确处理敏感数据？
  - 密码是否加密存储？
  - 日志是否包含敏感信息？
```

**示例：**
```javascript
// ❌ SQL 注入风险
const query = `SELECT * FROM users WHERE id = ${userId}`;

// ✅ 参数化查询
const query = 'SELECT * FROM users WHERE id = ?';
db.execute(query, [userId]);

// ❌ XSS 风险
element.innerHTML = userInput;

// ✅ 转义或使用 textContent
element.textContent = userInput;
```

### 6. 错误处理

```
✓ 异常捕获：
□ 是否捕获了可能的异常？
□ 是否给出清晰的错误信息？
□ 是否记录了错误日志？
□ 是否有合适的回退策略？

✓ 错误传播：
□ 是否吞掉了异常？
  ❌ try { ... } catch (e) { /* 什么都不做 */ }
□ 是否过度捕获？
  ❌ try { ... } catch (Exception e) { /* 太宽泛 */ }
```

**示例：**
```javascript
// ❌ 差的错误处理
try {
  await fetchData();
} catch (e) {
  // 吞掉异常，外部无法感知
}

// ✅ 好的错误处理
try {
  await fetchData();
} catch (error) {
  logger.error('Failed to fetch data:', error);
  throw new AppError('数据加载失败', { cause: error });
}
```

### 7. 测试覆盖

```
✓ 单元测试：
□ 核心逻辑是否有测试？
□ 边界条件是否有测试？
□ 错误场景是否有测试？

✓ 测试质量：
□ 测试是否独立？（不依赖执行顺序）
□ 测试是否可重复？（每次结果一致）
□ 测试是否清晰？（一看就懂在测什么）
```

---

### 8. 文档与注释

```
✓ API 文档：
□ 公共接口是否有文档？
  - 参数说明
  - 返回值说明
  - 异常说明
  - 使用示例

✓ README：
□ 如何运行？
□ 如何测试？
□ 如何部署？
```

## 审查流程

### 审查前准备

**作者自查：**
```
1. 运行测试（确保全部通过）
2. 运行 lint（确保代码规范）
3. 自己先读一遍代码
4. 补充必要的注释和文档
5. 提供测试截图或日志
```

### 审查中

**审查者检查：**
```
1. 理解需求（先看需求文档/Issue）
2. 查看改动范围（diff）
3. 逐文件审查（按上述清单）
4. 运行代码（本地验证）
5. 提出问题或建议
```

### 审查后

**反馈类型：**
```
🔴 必须修改（Blocker）
  - 功能错误
  - 安全漏洞
  - 严重性能问题

🟡 建议修改（Suggestion）
  - 命名改进
  - 代码优化
  - 注释补充

🟢 可选改进（Nit）
  - 代码风格
  - 微小优化
```

```

```

□ 测试覆盖
```

```
□ 所有8个维度

□ 考虑边界情况

## 总结

代码审查的目标：
1. **质量把关**：发现 Bug 和安全问题
2. **知识共享**：团队成员互相学习
3. **规范统一**：保持代码风格一致
4. **持续改进**：提升团队整体水平

审查态度：
- 对事不对人
- 提出建设性建议
- 解释你的理由
- 尊重作者的工作
