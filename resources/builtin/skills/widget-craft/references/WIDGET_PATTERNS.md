# Widget 模式分类与使用策略

PolarAgent 的 `WidgetType` 定义了 6 类 widget：`chart` / `form` / `table` / `mockup` / `diagram` / `interactive`。`widget-craft` 当前不再内置 HTML 模板，而是提供面向 `widget_code` 的模式判断与设计策略。选择实现前，先判断用户需求属于哪一类，再决定是直接写 `widget_code`，还是改用 Markdown / Mermaid 等更轻的载体。

## 1. 6 类 Widget 模式

| 类型 | 含义 | 典型场景 | 推荐实现 |
| --- | --- | --- | --- |
| `chart` | 数据可视化 | KPI、柱状、环形、趋势、分布 | `widget_code` + 内联 SVG / HTML |
| `table` | 结构化数据展示 | 数据表、清单、对比矩阵 | `widget_code` + `<table>` |
| `form` | 用户输入收集 | 表单、提单、配置项 | `widget_code` + 输入控件 + 事件桥 |
| `mockup` | 界面原型 | wireframe、布局示意 | `widget_code` |
| `diagram` | 关系 / 流程图 | 流程、组织、依赖图 | 优先 Mermaid；必要时再做 widget |
| `interactive` | 交互组件 | 看板、时间线、Tab、迷你工具 | `widget_code` + 明确交互模型 |

## 2. 模式选择原则

### chart

适合：

- 一个关键数字
- 多 KPI 并排
- 柱状对比 / 趋势
- 占比 / 分布

实现建议：

- 小型图表优先内联 SVG，不引第三方图库。
- 主系列用系统紫，辅系列用灰紫或低饱和衍生色。
- 图表不要追求花哨动画，重点是读数清楚。

### table

适合：

- 行列型数据
- 对比矩阵
- 项目 / 任务 / 明细清单

实现建议：

- 表头要压低成辅助语气，正文优先可扫读。
- 状态色要低饱和。
- 优先保证列宽和移动端换行可读，而不是塞太多装饰。

### form

适合：

- 提单
- 配置项
- 反馈收集

实现建议：

- label 明确、间距干净、操作区靠下。
- 输入实时上报用 `input`，提交动作用 `submit`。
- 保证 `patch` 模式下状态可恢复。

### mockup

适合：

- 原型框架
- 工具布局概念
- 工作台式 UI 草图

实现建议：

- 直接写 `widget_code`，按 `DESIGN_TOKENS.md` 约束出系统工具风格。
- 优先做“安静的工具界面”，不要做 landing page。

### diagram

适合：

- 需要解释关系 / 结构 / 流程的内容

实现建议：

- **优先 Mermaid 代码块**，不要为了能放进 widget 就强行 widget 化。
- 只有在需要点击节点、拖动、局部交互时，才考虑 `widget_code` 手写 SVG / HTML。

### interactive

适合：

- 任务流转
- 阶段切换
- 简易工具面板

实现建议：

- 每个动作都要有明确的事件语义。
- 可交互区域要完整，不要只让小图标响应。
- 交互组件仍然要维持工具型秩序感，不做游戏化装饰。

## 3. 选型决策树

```text
            用户想在对话里展示 / 交互某内容
                      │
        ┌─────────────┴─────────────┐
        ▼                           ▼
   是否是关系 / 流程图？        是否是结构化数据？
        │                           │
        ▼                           ▼
   优先 Mermaid                表格 / 清单 -> table

            否（不是上述）
                │
        ┌───────┴───────┐
        ▼               ▼
   是否是数据可视化？    是否需要用户输入？
        │               │
        ▼               ▼
      chart            form

   是否是原型 / 布局示意？   是否需要组件级交互？
        │                    │
        ▼                    ▼
     mockup             interactive

   都不是 -> 优先回到 Markdown，而不是硬做 widget
```

## 4. 推荐布局策略

### 单指标 / KPI

- 大数字优先，说明文字压低。
- 多指标网格尽量 2~4 项，不要一屏塞太满。

### 图表

- 柱状图适合趋势和对比。
- 环形图适合 3~5 类占比。
- 如果只是为了表达一行趋势，简单数字 + delta 往往比图更强。

### 数据表

- 有状态字段时，用低饱和 pill。
- 表头采用次要文字、轻表面。
- 行 hover 只做轻 tint。

### 表单

- 标题短，字段少，动作清晰。
- 不要把一个完整复杂流程塞进单个 widget。

### 时间线 / 看板

- 更像系统工具区块，不像展示海报。
- 让信息按顺序可扫读，别过度装饰节点。

## 5. 何时不用 Widget

以下场景更适合不用 widget：

- 单纯一段解释文字 -> Markdown
- 普通项目流程图 -> Mermaid
- 超长表格 / 超复杂表单 -> 先总结，再拆交互
- 需要真实持久化、后台联动的应用 -> 正式前端，不是会话 widget

## 6. 自检清单

- [ ] 选型是否真的属于 6 类之一。
- [ ] 有没有更轻的载体，比如 Markdown / Mermaid。
- [ ] 若用 `widget_code`，布局是否符合 Matte Cupertino / Apple Utility Pro。
- [ ] 交互语义是否清楚，是否全部走事件桥。
- [ ] 信息密度是否适合对话内浏览，而不是强行塞成复杂小应用。
