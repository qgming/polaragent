# 剧本与分镜写作

这项技能用于把一个想法推进成真正可写、可拍、可继续扩展的中文剧本资产。

对当前这类任务，不要把用户案例当成松散灵感。用户明确说“按照我给你的这些事例来写”时，`references/` 里的样例就是内置风格语料，必须先读后写。

## 何时使用

- 用户要写 `短片剧本`、`故事脚本`
- 用户要写 `短剧企划`、`剧本摘要`、`人物小传`、`故事梗概`
- 用户要写 `分镜剧本`、`镜头脚本`、`导演稿`
- 用户给出一个创意，要求扩写成完整剧情
- 用户给出已有剧本，要求按其样例风格重写
- 用户继续补充新的参考案例，要求存进技能中复用

## 输出模式

根据请求选择最贴近的一种，不要强行套统一格式。

### 模式 A：短片故事脚本

适用于 1-10 分钟短片、概念短片、情绪短片、类型短片。

默认结构：

```markdown
# 《片名》

时长：
类型：
一句话概念：

## 故事背景

## 风格核心

## 故事内核

## 人物设定

## 场景设定

## 道具设定

## 故事内容
```

要求：

- `故事背景` 要交代世界规则，不只写情绪。
- `风格核心` 要点出视觉反差、类型混搭或叙事气质。
- `故事内核` 要明确主旨，不要虚空拔高。
- `人物设定`、`场景设定`、`道具设定` 要可视化、可拍摄。
- `故事内容` 要有镜头感和推进感，不写成干瘪梗概。

### 模式 B：短剧项目包

适用于竖屏短剧、连载短剧、系列神话后传、男频升级流、女频情感向、反转爽剧等。

默认结构：

```markdown
# 《项目名》

原始创意：

## 剧本摘要
集数：
故事类型：
目标受众：
核心梗：
一句话故事：

## 人物小传

## 故事梗概

## 单集推进 / 阶段推进
```

要求：

- `原始创意` 保留用户脑暴感，但整理成清晰世界规则和冲突主轴。
- `剧本摘要` 要像能拿去内部立项的版本，简明但有卖点。
- `人物小传` 重点写：视觉形象、核心标签、身份背景、成长经历、性格特点、角色关系、成长弧线。
- `故事梗概` 要写出主线递进、阵营关系、反转与结局代价。
- 如果用户给了集数，默认按该集数思考节奏；如果没给，可以先做 3 幕或 5 阶段推进。

### 模式 C：场次分镜剧本

当用户说 `分镜剧本`，且参考案例本身是“按场编号 + 场内动作 + 台词 + 空镜”的写法时，优先使用这种格式，而不是机械改成镜头表。

默认结构：

```markdown
### 场1-1
日 / 夜  内 / 外  场景名
出场人物：角色A、角色B

△ 画面动作描述
△ 画面动作描述
角色A（情绪 / vo / os）：台词
△ 动作推进
【空镜：补充画面或收尾意象】
```

要求：

- 每场都要有清晰的动作推进和信息推进。
- `△` 段落是可拍的画面描述，不要写抽象评价。
- `vo`、`os` 只在确有必要时用，别滥用。
- `【空镜】` 用来收束情绪、强化意象或完成转场。
- 如果用户案例偏爽剧，就把每场结尾尽量写出钩子、打脸点、揭示点或立场升级。

### 模式 D：镜头级分镜

当用户明确要逐镜头拆解时，再用镜头表结构。

```markdown
## 镜头 01
时长：
景别：
机位 / 运镜：
画面内容：
角色动作 / 表演：
台词 / 旁白 / 字幕：
声音设计：
转场：
备注：
```

### 模式 E：联动输出

当用户要“从创意到分镜”的一整套内容时：

1. 先写 `短片故事脚本` 或 `短剧项目包`
2. 再从中派生 `场次分镜剧本` 或 `镜头级分镜`
3. 保持前后设定一致，不要二次发明人物和世界规则

## 工作流程

### 1. 先判定项目类型

优先判断是：

- 短片
- 连载短剧
- 单集分镜
- 故事梗概扩写
- 参考案例改写

### 2. 抽取约束

主动提炼这些信息：

- 类型与题材
- 时长或集数
- 目标受众
- 世界观与规则
- 主角团 / 阵营
- 核心冲突
- 爽点、反转点、结局调性
- 风格关键词
- 是否要偏项目文档、偏文学、偏可拍执行

若信息不全，但已经足够开写，优先根据参考样例补出合理假设，并在开头短句标注。

### 3. 先读参考文件，再决定细节密度

从 [references/README.md](references/README.md) 找到匹配样例并阅读。

当前内置参考：

- `references/zombie-cleaner-short-film.md`
  - 适合：概念短片、3 分钟左右短片、强世界观、强视觉反差、末日浪漫、黑色幽默、设定块丰富的写法。
- `references/mythic-short-drama-bible-and-storyboard.md`
  - 适合：长线短剧、神话后传、男频热血、三界权谋、反乌托邦结局、项目摘要 + 人物小传 + 场次分镜联动写法。

### 4. 对齐用户案例的“密度”而不是只抄表面格式

要继承的是：

- 世界规则先行
- 风格关键词明确
- 人物视觉化设定充分
- 场景和道具具有生产级细节
- 剧情推进带钩子和升级
- 结局有代价、有反噬或强画面收束

不要只学会标题，不学会力度。

## 写作原则

1. 优先写“能拍出来的东西”，不要只写概念。
2. 人物关系必须推动剧情，不能只列标签。
3. 每个核心场景最好兼顾：动作、信息、情绪、立场变化。
4. 类型爽点要具体，例如：打脸、反杀、揭黑幕、立宏愿、封印、背刺、终局反噬。
5. 文风可以浓，但不能空。
6. 当用户要“按案例来”，细节宁可足，不要缩成普通大纲。

## 处理新案例

当用户继续提供新的短片或短剧案例，并要求收进技能：

1. 在 `references/` 下新增对应 Markdown 文件。
2. 尽量完整保存原文，不擅自删减。
3. 在 `references/README.md` 追加该案例的用途说明。
4. 只有当新案例改变了输出结构或工作流时，才更新本 `SKILL.md`。

## 重要约束

不要声称“按用户案例写的”却没有先读对应 reference 文件。
